Project Prompt: AI-Powered Content Automation Pipeline

Overview
Build a self-contained web application (no n8n, no third-party automation tools) that monitors Google Alerts RSS feeds, clusters daily news using AI, generates a roundup blog post with a branded image, publishes it to a website, and cross-posts to LinkedIn and Twitter/X — all automatically on a daily schedule.

Core Features
1. RSS Feed Ingestion

Fetch one or multiple Google Alerts RSS feeds daily via cron
Parse feed items (title, link, source, published date, snippet)
Deduplicate items across days using URL hashing stored in DB
Handle empty feeds, malformed XML, and network failures gracefully

2. AI Clustering and Content Generation

Send all fetched items to Claude/OpenAI API
Cluster related stories and pick Top 5 for the day
Generate a single daily roundup blog post (title, intro, 5 story sections, conclusion)
Generate SEO meta title, meta description, and tags automatically

3. Image Generation

Use Playwright to render a branded HTML card template (headline + date + logo)
Screenshot it as a high-quality PNG — no DALL-E dependency
Store image in your media library (S3 or local)

4. Website Publishing

Publish the blog post via Payload CMS REST API or WordPress REST API
Store the returned blog post URL, post ID, and publish timestamp in DB
Support draft mode (review before publish) and auto-publish mode

5. Social Cross-Posting

After successful blog publish, post to LinkedIn and Twitter/X with:

Summarized text (AI-generated, platform-specific character limits)
Blog post URL
Image attachment (if platform supports it)


Store social post IDs and timestamps per platform

6. Dashboard (Admin UI)

View all daily runs — date, status, blog URL, social post links
See what was published, at what time, on which platform
Manual trigger option (run pipeline now)
Retry failed steps individually (re-generate, re-publish, re-post)
View raw fetched RSS items per day
Toggle auto-publish ON/OFF per platform


Database Schema (PostgreSQL)
rss_items         — id, feed_url, item_url (unique hash), title, snippet, source, fetched_at, processed
daily_runs        — id, run_date (unique), status, created_at, completed_at, error_log
blog_posts        — id, run_id, title, content, meta_title, meta_description, tags, image_path, cms_post_id, cms_url, published_at, status
social_posts      — id, blog_post_id, platform (linkedin/twitter), post_id, post_url, posted_at, status, error
alerts_config     — id, feed_url, label, is_active, created_at
app_settings      — key, value (stores tokens, toggles, config)

Auth and Security
Application Auth

Single admin login (no public signup) — email + password
JWT-based session with refresh token rotation
All dashboard routes protected behind middleware auth check
Rate limiting on login endpoint (max 5 attempts, 15 min lockout)

API Keys and Tokens Storage

All secrets (OpenAI key, LinkedIn token, X API keys, CMS API key) stored in environment variables only — never in DB plain text
If storing OAuth tokens in DB (LinkedIn refresh token) — encrypt using AES-256 before storing
Never expose API keys to frontend — all AI and social API calls go through your backend only

OAuth Token Management

LinkedIn access token expires in 60 days — store encrypted refresh token in DB, auto-refresh before expiry
X API uses OAuth 1.0a or OAuth 2.0 — store access + secret tokens encrypted
Alert via dashboard if any token is expired or about to expire (within 7 days)

Webhook / Endpoint Security

If exposing any endpoint for manual triggers — require Bearer token auth
CORS locked to your own domain only
All publish and post actions logged with timestamp and actor (cron vs manual)

Audit Log

Every action logged: RSS fetch, AI call, image gen, blog publish, social post, manual retries
Log includes: timestamp, action, status, response summary, error if any
Logs stored in DB and viewable in dashboard — never auto-deleted (archive after 90 days)

Pipeline Flow (Daily Cron)
1. Cron triggers at scheduled time (e.g. 8:00 AM)
2. Fetch all active Google Alerts RSS feeds
3. Deduplicate against previously processed URLs
4. If < 3 new items → skip run, log "insufficient content"
5. Send items to AI → cluster → pick Top 5 → generate blog post + social text
6. Render HTML image card → Playwright screenshot → save to storage
7. Publish blog post to CMS → receive post URL + ID → save to DB
8. Post to LinkedIn → save post ID + timestamp
9. Post to Twitter/X → save tweet ID + timestamp
10. Mark daily_run as completed → update dashboard

Edge Cases to Handle

Google Alerts feed returns 0 items — skip gracefully, log it
AI API timeout or error — retry once, then mark run as failed
CMS publish fails — do not post to social, mark for manual retry
Social post fails — blog is already live, just retry social independently
Duplicate run for same date — block at DB level (unique constraint on run_date)
Token expiry — surface warning in dashboard 7 days before

## Key File Responsibilities

| File | What it does |
|---|---|
| `lib/pipeline/index.ts` | Runs all 6 steps in order, stops on failure, writes run status |
| `lib/pipeline/fetcher.ts` | Parses RSS XML, hashes URLs, skips duplicates, saves new items |
| `lib/pipeline/ai.ts` | One API call — clusters, picks top 5, returns blog + social copy |
| `lib/pipeline/image.ts` | Spins up Playwright, loads news-card.html, screenshots to PNG |
| `lib/pipeline/publisher.ts` | POSTs to Payload or WP REST API, saves cmsPostId + cmsUrl |
| `lib/pipeline/linkedin.ts` | Posts using stored encrypted token, handles refresh if expired |
| `lib/pipeline/twitter.ts` | Posts using OAuth 1.0a keys from env |
| `lib/crypto/encrypt.ts` | AES-256-GCM — used only for OAuth tokens stored in DB |
| `lib/auth/middleware.ts` | Validates JWT on every dashboard request server-side |
| `api/cron/run/route.ts` | Entry point for scheduled runs — verifies CRON_SECRET header |
| `templates/news-card.html` | Plain HTML, injected with title/date via query params by Playwright |
